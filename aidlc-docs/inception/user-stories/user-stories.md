## ユーザーストーリー

---

### US-01: 第一印象を掴む

**ペルソナ**: P1 田中さん（採用担当）

> インターン採用担当として、ポートフォリオを開いた瞬間に「このエントリーは確認する価値がある」と判断できるようにしたい。なぜなら、限られた時間の中で多くの候補者を効率よくスクリーニングする必要があるから。

**シナリオ**

```
Given: 田中さんがエントリーシートに記載された URL をクリックする
When:  ポートフォリオのトップページが読み込まれる
Then:
  - 3 秒以内にページが表示される
  - ファーストビューに名前・キャッチコピー・プロフィール画像が表示される
  - 「何ができる人なのか」が一目でわかる
  - ナビゲーションで他セクションへ移動できる
```

**受け入れ基準**

- [ ] Hero セクションがスクロールなしでファーストビューに収まる
- [ ] Lighthouse Performance スコアが 90 以上
- [ ] ページタイトル（`<title>`）に氏名が含まれる
- [ ] ロードアニメーションが 1 秒以内に完了する

---

### US-02: スキルセットを確認する

**ペルソナ**: P1 田中さん（採用担当）

> インターン採用担当として、候補者の技術スタックを素早く把握し、自社プロジェクトへの適性を判断したい。なぜなら、技術ミスマッチによる採用失敗を防ぎたいから。

**シナリオ**

```
Given: 田中さんがポートフォリオのスキルセクションを閲覧している
When:  技術スタックの一覧を確認する
Then:
  - 使用技術がカテゴリ別（例: 言語 / フレームワーク / ゲーム開発）に整理されている
  - 習熟度または関与度が視覚的に表現されている
  - アイコン or バッジで技術名を直感的に認識できる
```

**受け入れ基準**

- [ ] 最低 3 カテゴリ以上でスキルが分類されている
- [ ] 各スキルにアイコンまたはバッジが付いている
- [ ] スマートフォンでも一覧が見やすく表示される

---

### US-03: 制作物から実力を確認する

**ペルソナ**: P1 田中さん（採用担当）

> インターン採用担当として、候補者が実際に何を作れるのか具体的に把握したい。なぜなら、スキル一覧だけでなく「実際に動くものを作れるか」を重視しているから。

**シナリオ**

```
Given: 田中さんが制作物セクションを閲覧している
When:  プロジェクトカードの一覧を見る
Then:
  - 各プロジェクトのタイトル・説明・使用技術・サムネイルが表示される
  - GitHub リポジトリまたはデモサイトへのリンクが設置されている
  - 詳細説明（制作背景・工夫した点）を確認できる
```

**シナリオ（詳細確認）**

```
Given: 田中さんが気になるプロジェクトカードを見つけた
When:  カードをクリックまたは展開する
Then:
  - プロジェクトの詳細説明（背景・課題・解決策）が表示される
  - 使用技術タグが一覧で確認できる
  - 外部リンク（GitHub / デモ）へ遷移できる
```

**受け入れ基準**

- [ ] プロジェクトカードに画像・タイトル・説明・技術タグ・リンクが含まれる
- [ ] 詳細表示（モーダル or 展開）が機能する
- [ ] リンクは新しいタブで開く（`target="_blank"` + `rel="noopener noreferrer"`）
- [ ] データ追加は JSON / MDX ファイルの編集のみで完結する

---

### US-04: GitHub でコードと活動を確認する

**ペルソナ**: P1 田中さん（採用担当）

> インターン採用担当として、候補者の GitHub プロフィールへ簡単にアクセスし、コードの書き方と継続的な活動を確認したい。なぜなら、ポートフォリオだけでなく日頃の開発習慣も評価基準にしているから。

**シナリオ**

```
Given: 田中さんがポートフォリオを閲覧している
When:  GitHub リンクを探す
Then:
  - フッターまたは専用セクションに GitHub リンクが設置されている
  - どのリンクがどのサービスか、アイコン等で一目でわかる
  - クリックで GitHub プロフィールへ直接遷移できる
```

**受け入れ基準**

- [ ] GitHub・その他 SNS リンクがページ内に配置されている
- [ ] 各リンクにサービスロゴ / アイコンが表示される
- [ ] 新しいタブで開く

### US-05: 新しい実績を追加する

**ペルソナ**: P2 Soma（オーナー）

> サイトオーナーとして、新しいプロジェクトが完成したとき、コンポーネントを修正せずにデータファイルを追加するだけで実績を反映させたい。なぜなら、更新コストが高いとサイトが古くなって放置されてしまうから。

**シナリオ**

```
Given: 新しいプロジェクトが完成し、ポートフォリオに追加したい
When:  プロジェクトデータファイル（JSON または MDX）を作成してリポジトリに追加する
Then:
  - GitHub Actions が自動でビルド・デプロイを実行する
  - 制作物セクションに新しいプロジェクトカードが表示される
  - コンポーネントのコードを一切変更する必要がない
```

**受け入れ基準**

- [ ] プロジェクトデータは 1 ファイル（JSON または MDX）の追加で反映される
- [ ] データスキーマが README に文書化されている
- [ ] GitHub Actions による自動デプロイが設定されている
- [ ] デプロイ完了まで 5 分以内

---

### US-06: Phase 2 コンテンツを後から追加する

**ペルソナ**: P2 Soma（オーナー）

> サイトオーナーとして、インターン選考が落ち着いたあとに職歴・記事リンクを追加し、ポートフォリオを育て続けたい。なぜなら、サイトは一度作ったら終わりではなく、キャリアとともに成長させるものだから。

**シナリオ**

```
Given: インターン選考が一段落し、Phase 2 コンテンツを追加したい
When:  職歴セクション（FR-05）または記事リンクセクション（FR-06）を有効化する
Then:
  - 既存のデザイン・レイアウトと統一感が保たれている
  - 他セクションへの影響なく新セクションが追加できる
  - ナビゲーションに新セクションが自動で追加される
```

**受け入れ基準**

- [ ] セクションの追加・非表示が設定ファイルで制御できる
- [ ] Phase 2 セクションのコンポーネントが事前にスケルトン実装されている

---

### US-07: SNS 経由でプロフィールを確認する

**ペルソナ**: P3 鈴木さん（技術仲間）

> 技術コミュニティの仲間として、X や GitHub 経由でたどり着いたポートフォリオで、この人の技術スタックと何を作っているかをざっと把握したい。なぜなら、技術的に共鳴できる相手かどうかを素早く判断したいから。

**シナリオ**

```
Given: 鈴木さんがスマホで X のタイムラインを眺めていて、
       Soma のポートフォリオ URL を見かけた
When:  スマホのブラウザでポートフォリオを開く
Then:
  - モバイルで崩れずに表示される
  - スクロールで Hero → スキル → 実績の順に確認できる
  - 「この人は〇〇系のエンジニアだ」と 1 分以内に把握できる
```

**受け入れ基準**

- [ ] 320px 幅のモバイル端末で全セクションが正常に表示される
- [ ] タップ操作でカードの詳細が確認できる
- [ ] ページ全体のスクロールが快適に機能する

---

### US-08: 気になったプロジェクトの GitHub を見に行く

**ペルソナ**: P3 鈴木さん（技術仲間）

> 技術コミュニティの仲間として、ポートフォリオで気になったプロジェクトのコードをすぐに GitHub で確認したい。なぜなら、実装の詳細や技術選定の意図を知りたいから。

**シナリオ**

```
Given: 鈴木さんが制作物セクションで気になるプロジェクトを見つけた
When:  GitHub リンクをタップ / クリックする
Then:
  - 対象リポジトリの GitHub ページが新タブで開く
  - ポートフォリオのページはそのまま残る
  - リポジトリが存在する場合は正しい URL に遷移する
```

**受け入れ基準**

- [ ] 各プロジェクトカードに GitHub リポジトリへのリンクが設定できる（任意）
- [ ] リンクが無いプロジェクトは「非公開」等のラベルを表示する
- [ ] 新タブで開く（`target="_blank"` + `rel="noopener noreferrer"`）
